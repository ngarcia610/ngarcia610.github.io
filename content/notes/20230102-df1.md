This is the first of five courses that establishes a foundational knowledge of DevOps theories.

## DevOps Basics

The Five Levels of Devops

1. Values - CAMS model
2. Principles - the 3 ways
3. Methods - the 5 methodologies
4. Practices
5. Tools

CAMS model

- Culture
- Automation
- Measurement
- Sharing

The 3 ways

1. Systems Thinking
2. Amplifying feedback loops
3. Continuous experimentation and learning

Methodologies

1. People > Process > Tools
2. Continuous Delivery
3. Lean Management: small batches, WIP limits, Feedback loops, visualization
4. Change control
5. Infrastructure as Code: systems treated like code, checked into source control, reviewed built and tested

Practices

1. Incident Response
2. Developers on call
3. Public status pages
4. Blameless Postmortems
5. Embedded Teams
6. Cloud Solutions
7. Andon Cords - Anyone can stop production in an emergency
8. Dependency Injection - applications should not be aware of their external dependencies (loosely coupled)
9. Blue/Green Deployment - two identical systems with a load balancer
10. Chaos Monkey - intentionally creating faults to build resiliency

Tool Criteria

1. Programmable
2. Verifiable
3. Well behaved

## DevOps Culture

Blameless postmortems

- Requirements:
  - Do it within 48 hours
  - Elect a leader (not the person who caused the outage)
- Document example:
  - Description of the incident
  - Description of root cause
  - How was the incident stabilized or fixed
  - Timeline of events and actions taken
  - How were customers affected
  - Recommended improvements

Management best practices

1. Independent, cross-functional teams
2. Help people through change
3. Use agile, lean processes

## Buiding Blocks of DevOps

Kaizen's Guiding Principles

1. Good processes bring good results
2. Go see for yourself
3. Speak with data and manage by facts
4. Take action to contain and correct root causes
5. Work as a team
6. Kaizen is everybody's business

[Kaizen cycle](https://cdn.ttgtmedia.com/rms/onlineimages/erp-kaizen_cycle.png)

Principles of Lean Software

1. Eliminate Waste
2. Amplify learning
3. Decide as late as possible
4. Decide as fast as possible
5. Empower the team
6. Build in integrity

Types of waste

1. Muda - work that absorbs resources, but adds no value
2. Muri - unreasonable work imposed on workers and machines
3. Mura - work coming in unevenly instead of a constant or regular flow
4. Partially done work
5. Extra features
6. Relearning
7. Handoffs
8. Delays
9. Task switching

Build, Measure, Learn

1. Build the minimum viable product.
2. Measure the outcome and internal metrics.
3. Learn about your problem and your solution.
4. Repeat. Go deep where it's needed.

ITIL phases

1. Service strategy
2. Service design
3. Service transition
4. Service operation

## CD and CI

Deployment Definitions:

1. Provisioning - making a server ready for operation (hardware, OS, services, network)
2. Deployment - automatically deploying and upgrading applications on a server
3. Orchestration - coordinated operations across multiple systems
4. Configuration Management - management of change control for system configuration after initial provision; maintaining and upgrading the application and its dependencies
5. Imperative (procedural) - commands necessary to produce a desired state are defined and executed.
6. Declarative (functional) - a desired state is defined, relying on the tool to configure a system to match that state.
7. Idempotent - the ability to execute repeatedly, resulting in the same outcome
8. Self-service - the ability for an end user to initiate a process without having to go through other people

Continuous Delivery Definitions:

1. Continuous Integration - automatically building and unit testing the entire application
2. Continuous Delivery - deploying every change to a production-like environment, and performing automated integration and acceptance testing
3. Continuous Deployment - after testing every change, automatically deploying to production

Continuous Delivery Practices

1. Builds should be less than 5 minutes
2. Commit small bits
3. Don't leave the build broken
4. Use a trunk-based development flow (no long running branches, multiple commits daily)
5. Don't allow flaky tests
6. The build should return a status, a log, and an artifact

Continuous Delivery Pipeline

1. Only build artifacts once
2. Artifacts should be immutable
3. Staging should be a copy of production
4. Stop deploys if a previous step fails
5. Deployments should be idempotent

Types of testing

1. Unit testing (function within the program to test)
2. Code hygiene (linting, formatting, banned code)
3. Integration testing (same as unit testing but includes all apps, components, dependencies in a test environment)
4. TDD, BDD, ATDD
    - TDD: states desired outcome as a test
    - BDD: work with stakeholders
    - ATDD: scenarios are from the end users' perspective
5. Infrastructure testing
6. Performance testing
7. Security testing

CI Tools

1. Version control: github, bitbucket
2. CI systems: Jenkins
3. Build: Make/Rake, Maven, Gulp, Packer
4. Test: JUnit, Linters, Robot, Vagrant, Otto, Docker compose
5. Artifact repo: Artifactory, Nexus, Docker Hub, Tags with S3
6. Deployment: Rundeck, Urbancode, Thoughtworks
