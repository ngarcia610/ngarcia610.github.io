## Boot, reboot, shutdown

- sudo systemctl reboot
- sudo systemctl poweroff
- Use `--force or --force --force` for emergencies
- Use `shutdown 02:00` for scheduled reboots
  - 24 hour time format
  - You can also use `+15` to shutdown in 15 min
  - Use `-r` to schedule a reboot
  - You can add a wall message with single quotes at the end
- Cancel a pending shutdown with `-c`

## Boot Targets

- Use the `get-default` subcommand to view the current boot target
- Use the `set-default` subcommand to set the boot target (requires reboot)
- Use the `isolate` subcommand to start one unit and stop all others
- Example: `sudo systemctl set-default multi-user.target`
- `sudo systemctl isolate graphical.target` no reboot
- `sudo systemctl isolate emergency.target` recovery mode
- `sudo systemctl isolate emergency.target` root shell to fix system settings
- Root password needs to be set in order to use these targets

## Common systemd targets

- rescue: unit target that pulls in the base system and spawns a rescue shell
- multi-user: unit target for setting up a multi-user system
- graphical: unit target for setting up a graphical login screen
- emergency: unit target that starts an emergency shell on the main console

## Install, configure, and troubleshoot bootloaders

- Example is on Centos 8
- Boot from installation media
- From startup select `troubleshoot`
- From rescue select option 1 for continue
- The system is then mounted under /mnt/sysroot
- Use `chroot /mnt/sysroot` to make this active
- Generate a new grub config with `grub2-mkconfig -o /boot/grub2/grub.cfg`
- If you're using an EFI system `grub2-mkconfig -o /boot/efi/EFI/centos/grub.cfg`
- Next use `lsblk` to view attached disks. Check where `/` is installed.
- Install at that location with `grub2-install /dev/sda`
- For EFI make sure you have the correct packages installed `dnf resinstall grub2-efi grub2-efi-modules shim`
- Type exit twice to leave the shell and rescue environment
- `sudo vim /etc/default/grub` edit GRUB_TIMEOUT= or GRUB_CMDLINE_LINUX=""
- `sudo grub2-mkconfig -o /boot/grub2/grub.cfg` again after changes

## Use Scripting to Automate Tasks

- `.sh` file extension is a convention used to identify script files but isn't needed.
- The first line of a script uses the shebang `#!/bin/bash`
- Set the permission of the file to make it executable.
- Use `chmod u+x script.sh` for only the owner, or `+x` for all.
- A good reference file can be found at `/etc/cron.hourlv/0anacron`

Example script using logic:

```bash
#!/bin/bash

if test -f /tmp/archive.tar.gz; then
    mv /tmp/archive.tar.gz /tmp/archive.tar.gz.OLD
    tar acf /tmp/archive.tar.gz /etc/dnf/
else
    tar acf /tmp/archive.tar.gz /etc/dnf/
fi
```

## Manage the startup process and services (In Services Configuration)

- View service details: `systemctl cat sshd.service`
- Modify service files: `sudo systemctl edit --full sshd.service`
- Revert a service file to defaults: `sudo systemctl revert sshd.service`
- View the status of a service: `sudo systemctl status sshd.service`
- Control services with `stop`, `start`, `restart`, `reload`
- Control if a service starts at boot with `enable`, `disable`, or view with `is-enabled`
- Brute force prevent a service from starting with `mask` and cancel with `unmask`
- List all units regardless of status: `sudo systemctl list-units --type service --all`

## Create Systemd Services

- `/usr/local/bin` is a common directory used for user applications
- `man systemd.service` to show unit configuration options
- A restart loop occurs when a service crashes and immediately tries to restart again only to crash. You can prevent this by adding a cooldown period with `RestartSec=`
- Systemd service files must have at least 3 sections:
  - `[Unit]`
  - `[Service]`
  - `[Install]`
- You can copy an existing service file to use as a template
- `sudo cp /lib/systemd/system/ssh.service /etc/systemd/system/new.service`
- `After=` tells systemd when the service should start.
- `ExecStart=` needs to point to the path of the shell script
- Whenever you modify a systemd service, you need to reload: `sudo systemctl daemon-reload`
- `sudo journalctl -f` allows you to view logs in realtime

## Diagnose and Manage Processes

- `ps -aux` is used to view all processes with user columns for formatting
- `ps -lax` for all columns with nice values
- ps can use '-' or no dash with options (UNIX versus BSD style) each with different output
- View memory and CPU usage by PID with `ps u 1` where 1 is the PID
- `top` shows processes and reorders them based on cpu usage
- You can just process priority by changing the nice value.
- `nice -n 11 bash` assigns 11 nice value to the bash process
- The lower the nice value, the higher the priority
- Users can set nice from 0-19, only root can set negative numbers
- Example: Change the nice value of ssh `sudo renice 9 996` (9 is the nice value, 996 is the PID)
- `ps l` will show nice values for processes
- Use `lsof` to list files opened by a process
- Example: show files opened by PID 1 `sudo lsof -p 1`
- `kill -L` will list all possible options for terminating a process
- `sudo kill -SIGHUP 1147` kills PID 1147
- `pgrep -a bash` shows PID for bash
- You can use ctrl + z to pause a program and then use `fg` to bring it back to the foreground
- You can "background" a program by adding `&` to the end
- `sleep 10` would initiate sleep for 10 seconds

## Locate and Analyze System Log Files

- Logging daemons collect, analyze, and store logs (text files about status, error, and warning messages)
- `rsyslog` is a common logging daemon that stores logs in `/var/log` directory
- If you don't know where logs are stored, you can grep for a keyword in `/var/log`
- Example: `grep -r 'ssh' /var/log` will show logs stored in `/var/log/secure`
- `journalctl` allows you to analyze logs more efficiently
- Examples:
  - Find the location of the sudo command `which sudo` returns `/bin/sudo`
  - Next show only logs from that command `journalctl /bin/sudo`
- Journactl alone opens all logs. `journalctl -e` goes to the end when opening.
- Log types: info, warning, err, crit are the common ones. You can view all of them with `journalctl -p` and tab x2.
- Boot logs can be viewed with `journalctl -b 0` or `journalctl -b -1`
- Use `last` command to view who logged into the system. Newest entries are at the top.
- The `lastlog` command can show the time a user last logged in, and in remote sessions the IP is saved in the 'From" column.
- `sudo journalctl --unit=sshd.service -n 20 --no-pager` to find what IP address last connected successfully
- Example of using regex in journalctl `sudo journalctl -p info -g '^c'`

## Schedule tasks to run at a set date and time

Schedule jobs with cron

- `cat /etc/crontab`
- The first 5 values tell when the job should run (minute, hour, day of month, month, day of week)
- The next value is a username
- Next is the command to run with options
- Additional characters
  - `*` = match all possible values (i.e., every hour)
  - `,` = match multiple values (i.e., 15,45)
  - `-` = range of values (i.e., 2-4)
  - `/` = steps (i.e., */4)
- Always use the full path to a command for cron jobs
- `crontab -e` edit the table of the current user you're logged in as
- `crontab -l` to see the current user's crontab or add sudo to see root
- `sudo crontab -e -u user2` to edit another user's crontab
- `crontab -r` to remove the crontab
- There are default directories for cron jobs:
  - `/etc/cron.daily/`
  - `/etc/cron.hourly/`
  - `/etc/cron.monthly/`
  - `/etc/cron.weekly/`

Schedule jobs with anacron

- `sudo vim /etc/anacrontab`
- Format is as follows:
  - period in days
  - delay in minutes
  - job id
  - command
- Example: `10 5 db_cleanup /usr/bin/touch /root/anacron_created_this`
  - runs once every 10 days
  - 5 minutes of delay
  - Job id is db_cleanup
  - Command to run is `/usr/bin/touch /root/anacron_created_this`

Schedule jobs with at

- Used to schedule jobs that do not run on a regular schedule
- `at -V` to verify it is installed
- `at` schedule a task
- `atq` list jobs for current user, or sudo for all users
- `atrm` removes a job by job number
- `batch` only run the job if the system load meets the specified requirement
- The format is `at [time] [date/day]`
- ctrl + d to save the job
- ctrl + a to view all commands

## Verify completion of scheduled jobs

- `sudo cat /var/log/cron` to verify cron and anacron jobs ran
- `sudo grep CMD /var/log/cron`
- `sudo anacron -n` force run all the jobs scheduled for today or add `-f` to force jobs that ran already
- At the end of your anacrontab line you can add `| systemd-cat --identifier=test`
- That utility logs everything
