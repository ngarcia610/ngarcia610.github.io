## Testing locally with docker

Docker virtualizes the operating system kernel and provides a runtime for installing and running applications in the same way anywhere.

Installation on MacOS

- <https://brew.sh> to find the following command `/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"`
- This is used to install the homebrew package manager
- Once installed run `brew cask install docker`
- This installs the docker client as well as the docker daemon
- Run docker and enter password to elevate
- `docker run hello-world` is the test application

Writing the dockerfile

- <https://hub.docker.com> is a repository of docker images
- Use `FROM scratch` to use a completely blank image (usually you use something that exists already)
- `vim Dockerfile`
- Alpine linux is a small linux distro focused on security. Good for containers since it is small that gives more room for images.

```yaml
FROM nginx:alpine
MAINTAINER Nate Garcia <me@nathangarcia.net>

COPY website /website
COPY nginx.conf /etc/nginx/nginx.conf

EXPOSE 80
```

- FROM line is choosing the image and tags from docker hub
- MAINTAINER line is showing who owns and updates this file
- COPY line copies new files or directories from the source and adds them to the destination on the filesystem of the container
- EXPOSE is controlling ports so that the local system can access the container
- <https://docs.docker.com/engine/reference/builder/> is the official documentation for the dockerfile
- `docker build --tag website .` to build the image
- `docker run --publish 80:80 website` to start the container; the publish flag uses `<container port>:<host port>`
- Use `docker images` to view images you have locally

Writing the docker compose manifest

- Docker compose allows you to run multiple containerized applications within a single network, and to be lazy by putting all of your common Docker commands in one place.
- `vim docker-compose.yml`

```yaml
version: '3.7'
services:
  website:
    build:
      context: .
  ports:
    - 80:80
```

- `docker-compose up website` initialize the network and services then build the website
- Find existing containers running within docker-compose `docker-compose ps`
- Take the site down `docker-compose down`

## Testing your app with rspec, capybara, and selenium

- rspec - ruby based testing framework
- capybara - a webdriver (spins up the browser and runs tests against it) that allows you to interact with the website
- `mkdir -p spec/unit`
- `vim spec/unit/page_spec.rb`

```yaml
require 'capybara'
require 'capybara/dsl'

describe "Example page render unit tests" do
  it "Should show the Company logo" do
  end
end
```

- Update the docker compose

```yaml
version: '3.7'
services:
  website:
    build:
      context: .
  ports:
    - 80:80
  unit-tests:
    volumes:
      - "$PWD:/app"
    build:
      context: .
      dockerfile: rspec.dockerfile
    command:
      - --pattern
      - /app/spec/unit/=_spec.rb
```

- Make the dockerfile that was referenced in the docker compose
- `vim rspec.dockercompose`

```yaml
FROM ruby:alpine
MAINTAINER Nate Garcia <me@nathangarcia.net>

RUN apk add build-base ruby-nokogiri
RUN gem install rspec capybara selenium-webdriver

ENTRYPOINT [ "rspec"]
```

- Docker containers start with /bin/sh -c
- CMD line lets you provide additional commands to that shell
- ENTRYPOINT line in the above dockerfile runs 'rspec' alone
- The difference between CMD and ENTRYPOINT is how arguments are handled
- CMD applies arguments to sh

Running the test

- `docker-compose up -d website` the d flag suppresses output
- `docker-compose run --rm unit-tests` the rm option specifies to not save the container when done
- This can take a while

Improving the test

- Validate your docker compose file with `docker compose config`
- Chrome Developer tools allows you to isolate the element you want to inspect.
- We want to link selenium to capybara
- Since chrome doesn't run in the container, we specify remote
- We also add some additional chrome options for quality of life

```yaml
require 'capybara'
require 'capybara/dsl'
require 'selenium-webdriver'

include Capybara::DSL
Capybara.app_host = "http://website"
Capybara.register_driver :selenium, do |app|
  Capybara::Selenium::Driver.new(app,
    :browser => :remote,
    :url => "http://#{ENV['SELENIUM_HOST']:#{ENV['SELENIUM_PORT']}/wd/hub"
    :desired_capabilities => Selenium::WebDriver::Remote::Capabilities.chrome(
      "chromeOptions" => { "args" => [
        ' -no-default-browser-check',
        '--disable-dev-shm-usage'
      ] }
    )
  )
end
Capybara.default_driver = :selenium

describe "Example page render unit tests" do
  it "Should show the Company logo" do
    visit('/')
    expect(page.has_selector? 'a.logo').to be_true
  end
end
```

- Add selenium service to docker compose
- Selenium automates browser activity
- The image selenium standalone-chrome-debug includes a vnc server to see the testing.
- The VNC ports are 5901 (inside) 5900 (outside)

```yaml
version: '3.7'
services:
  selenium:
    image: selenium/standalone-chrome-debug
    ports:
      - 4444:4444
      - 5901:5900
  unit-tests:
    environment:
      SELENIUM_HOST: selenium
      SELENIUM_PORT: 4444
    volumes:
      - "$PWD:/app"
    build:
      context: .
      dockerfile: rspec.dockerfile
    command:
      - --pattern
      - /app/spec/unit/*_spec.rb
  website:
    build:
      contect: .
    ports:
      - 80:80
```

- `docker-compose up -d`

## IaC with Terraform

Create the Terraform Dockerfile

- Terraform is a cross platform deployment tool written in Golang
- <https://terraform.io>
- `vim terraform.dockerfile`

```yaml
FROM alpine
MAINTAINER Nate Garcia <me@nathangarcia.net>

RUN wget -O /tmp/terraform.zip https://releases.hashicorp.com/terraform/1.3.9/terraform_1.3.9_linux_amd64.zip
RUN unzip /tmp/terraform.zip -d /

USER nobody
```

- `docker build -t terraform . -f terraform.Dockerfile`
- Verify the terraform file was inflated
- `docker run --rm --interactive --tty --entrypoint sh terraform`
- This allows you to run commands through the docker container
- `exit` to leave

- `docker build --tag terraform --file terraform.Dockerfile .`

Writing Terraform code

- `vim main.tf`

```terraform
data "aws_iam_policy_document" "bucket_policy" {
  statement {
    sid = "PublicReadGetObject"
    effect = "Allow"
    actions = [ "s3:GetObject" ]
    principals {
      type = "*"
      identifiers = [ "*" ]
    }
    resources = [ "arn:aws:s3:::explorecalifornia.org/*" ]
  }
}

/* We can access properties from data sources using this format:
   ${data.<data_source_type>.<data_source_name>.<property>.

   In this case, we need the JSON document, which the documentation
   says can be accessed from the .json property. */

resource "aws_s3_bucket" "website" {
  bucket = "explorecalifornia.org"  // The name of the bucket.
  acl    = "public-read"            /* Access control list for the bucket.
                                       Websites need to be publicly-available
                                       to the Internet for website hosting to
                                       work. */
  policy = "${data.aws_iam_policy_document.bucket_policy.json}"
  website {
    index_document = "index.htm"   // The root of the website.
    error_document = "error.htm"   // The page to show when people hit invalid pages.
  }
}

output "website_bucket_url" {
  value = "${aws_s3_bucket.website.website_endpoint}"
}
```

Deploy the infrastructure with Terraform

- `docker-compose run --rm terraform init`
- Terraform keeps track of deployed infrastructure in a file called terraform.tfstate
- Losing the state file can create a lot of problems.
- The state file needs to be private (because of stored credentials and confidential information), but also sync'd so that others can contribute to infrastructure deployments.
- Terraform backends fix this problem. S3 is a common solution to use as a backend.
- <https://developer.hashicorp.com/terraform/language/settings/backends/configuration>
- Add TLS for terraform.dockerfile

```yaml
FROM alpine
MAINTAINER Nate Garcia <me@nathangarcia.net>

RUN wget -O /tmp/terraform.zip https://releases.hashicorp.com/terraform/1.3.9/terraform_1.3.9_linux_amd64.zip
RUN unzip /tmp/terraform.zip -d / && apk add ca-certificates

USER nobody
ENTRYPOINT ["/terraform"]
```

- `docker-compose build terraform`
- Verify the changes with `docker-compose run --rm terraform init`
- It will prompt for an availability zone, you can choose what's closest to you such as us-east-1.
- For aws credentials we want to include them in the docker-compose.yml under 'environment'

```yaml
version: '3.7'
services:
  terraform:
    build:
      context: .
      dockerfile: terraform.Dockerfile
    environment:
      AWS_ACCESS_KEY_ID: <USERID>
      AWS_SECRET_ACCESS_KEY: <SECRETKEY>
      AWS_REGION: us-east-1
      AWS_DEFAULT_REGION: us-east-1
  selenium:
    image: selenium/standalone-chrome-debug
    ports:
      - 4444:4444
      - 5901:5900
  unit-tests:
    environment:
      SELENIUM_HOST: selenium
      SELENIUM_PORT: 4444
    volumes:
      - "$PWD:/app"
    build:
      context: .
      dockerfile: rspec.dockerfile
    command:
      - --pattern
      - /app/spec/unit/*_spec.rb
  website:
    build:
      contect: .
    ports:
      - 80:80
```

- Stage the deployment with `docker-compose run --rm terraform plan` type 'yes' when prompted
- Create the infrastructure with `docker-compose run --rm terraform apply`

## CI/CD with Jenkins

Installation

- Continuous Integration allows to run tests on every commit to reduce the chances of a bad build.
- Groovy is a java based scripting language that will allow us to automate Jenkins
- For plugins visit <https://plugins.jenkins.io>
- Jenkins default port is 8080
- `vim jenkins.dockerfile`

```yaml
FROM jenkins/jenkins:alpine
MAINTAINER Nate Garcia <me@nathangarcia.net>

COPY plugins.txt /usr/share/jenkins/plugins.txt
RUN /usr/local/bin/install-plugins.sh < /usr/share/jenkins/plugins.txt
EXPOSE 8080
```

- plugins.txt file

```yaml
workflow-aggregator
seed
git
```

- `docker-compose up jenkins` this can take a while on the first run

- /var/jenkins_home is where Jenkins configuration and state is stored.

Create a Jenkinsfile `vim Jenkinsfile`

```yaml
pipeline {
  agent any
  stages {
    stage('Build our website'){
      steps {
        sh "scripts/build.sh"
      }
    }

    stage('Run unit tests'){
      steps {
        sh "scripts/unit_tests.sh"
      }
    }

    stage('Deploy website'){
      steps {
        sh "scripts/deploy_website.sh"
      }
    }
  }
}
```

Using Jenkins to deploy the app

- `git init`
- `git checkout -b master`
- `git add -A`
- `git commit -m "Initialize repo." .`
- `docker-compose up jenkins`
- Once Jenkins is up, login, use underscores instead of spaces for the item name, and select multibranch pipeline
- Fill in the general information
- Run the tests. Blue means it passed, red means it failed, check the logs for more info on troubleshooting.
